## Story

As a developer,
I want events + event_rounds + groups + group_members + rule_sets + rule_set_revisions + invites + sub_games + sub_game_participants + device_bindings tables defined, plus the `players` table extended with full identity columns beyond T1-6a's minimal slice,

## Naming convention (read first)

Throughout this spec, table + column names follow Drizzle's standard pattern: **Drizzle property names are camelCase, SQL column names are snake_case** (one-to-one mapping). For example, `holesToPlay: integer('holes_to_play')` declares a Drizzle property `holesToPlay` mapped to a SQL column `holes_to_play`. The migration file (0002 SQL) contains the snake_case names; tests + application code reference the camelCase Drizzle properties.

**ACs reference SQL column names** for constraints, indexes, and migration assertions (because those are what `drizzle-kit generate` emits). Drizzle property names appear in code-shape examples. The two are interchangeable in meaning — only the form context differs.

**Audit step for the existing `players.google_sub` column** (AC #4 / Subtask 5.1): the dev MUST grep the existing `apps/tournament-api/src/db/schema/players.ts` for both `'google_sub'` (the SQL column name) AND `googleSub` (the Drizzle property). If either is present, do NOT add a duplicate column or a duplicate partial unique index — that would crash `drizzle-kit generate`.

### 3. Tenant + context_id stamping (FD-6)

Every new table inherits `tenantId` + `contextId` via `ecosystemColumns()` from `_columns.ts` (matches T1-6a + T2-1 posture):
- `tenantId` defaults to `'guyan'` (single-tenant v1)
- `contextId` is NOT NULL with no default — the application code must stamp it explicitly per FD-6

**Stamping rules** (per epic line 858-860, baked into the spec for downstream code reference):
- `events.context_id = 'event:' + events.id` — opaque, generated before insert
- `event_rounds`, `groups`, `group_members`, `invites`, `device_bindings`, `sub_games`, `sub_game_participants` inherit their parent event's `context_id`
- `rule_sets` + `rule_set_revisions` use `'library:{tenant_id}'` (parallel to courses' `'library:guyan'`) — these are tenant-scoped rule libraries that span events

This story does NOT enforce stamping at runtime (no triggers, no app-code yet); the application code in T3-2/T3-5/T3-6 is responsible for populating `contextId` correctly. Schema-level enforcement is just the NOT NULL constraint via `ecosystemColumns()`.

### 4. Players table extension is non-destructive

Per epic line 848, columns added to existing `players` table:
- `name TEXT NOT NULL` — but existing T1-6a rows have no name. To make the migration non-destructive, the column is added as `NOT NULL DEFAULT ''` so existing rows get an empty string. Application code (T3-2 wizard, T3-3 group CRUD) is responsible for filling real names.
- `ghin TEXT` (nullable) + partial unique index `WHERE ghin IS NOT NULL`
- `google_sub TEXT` (nullable) + partial unique index — duplicates the column added in migration 0025 of Wolf Cup but for tournament-api it's NEW (T1-6a's tournament players doesn't have it). Confirm at impl time whether T1-6a already added it.
- `apple_sub TEXT` (nullable) + partial unique index — v1.5 use only, schema-ready
- `manual_handicap_index REAL` (nullable) — for non-GHIN players or GHIN outage fallback
- `preferred_tee_color TEXT` (nullable)

**The `name` default-empty-string compromise** is acceptable because:
- Production state: NO production rows exist yet (the tournament-api DB is empty pre-T3 launch).
- If T1-6a had OAuth-bound players, they'd get name='' which is benign — the OAuth callback still works because `name` isn't read on auth path.
- T3-2 wizard rejects empty names client-side; T3-3 group CRUD rejects empty names server-side. So no path lets an empty name persist after T3-2/T3-3 ship.

### 5. device_bindings.session_id is NULLABLE (T3-6 dependency)

Per epic line 852, `device_bindings(id PK, player_id FK, session_id FK NULLABLE, device_info, created_at)`. The NULLABLE is **load-bearing** for T3-6's invite-link "that's me" claim flow:
- Pre-SSO: organizer creates invite → guest claims `player_id` on a device → row inserted with `session_id = NULL`
- Post-SSO (T3-7): the same device's session is created → `device_bindings.session_id` is updated to point at the new sessions row

If `session_id` were NOT NULL, T3-6 couldn't model the pre-SSO claim state. The migration MUST allow NULL.

### 6. sub_games FK targets event_rounds, NOT scoring rounds

Per epic line 856: `sub_games(id PK, event_round_id FK → event_rounds.id, ...)`. **Critical** because:
- `event_rounds` is the SETUP-time entity (T3-1, this story).
- `rounds` (the SCORING-time entity, T5.1) doesn't exist yet.
- T6.13's dispatcher joins `sub_games` via `event_round_id` to the scoring `rounds` row at compute time.

If sub_games FK'd to scoring `rounds`, T3-1 would have a forward dependency on T5.1 — broken layering. The current shape lets T3-9 (sub-game opt-in UI) ship in T3 sequence with no T5 dependency.

Same for `rule_set_revisions.effective_from_round_id FK → event_rounds.id` (per epic line 844): mid-event rule edits reference the SCHEDULE entity, not the scoring entity.

### 7. invites is event-scoped only (no per-player invites in v1)

Per epic line 836: `invites(id PK, event_id FK, token TEXT UNIQUE, expires_at, created_by_player_id FK, created_at)`. NO `player_id` column — per-player invites are a v1.5+ feature. The token is event-wide; whoever clicks the link can claim any open player slot.

### 8. holes_to_play locked at event_round creation (no v1 mutation path)

Per epic line 836: `event_rounds.holes_to_play INTEGER NOT NULL DEFAULT 18 CHECK(holes_to_play IN (9, 18))`. No update path planned for v1. Supports 9-hole rounds (Emergency 9, Member-Member-style two-9-match days, Fall 9-hole tournaments). All 4 Pinehurst rounds default to 18.

Multiple `event_rounds` can share a `round_date` (no UNIQUE constraint) — enables 27-hole days (18 + 9) via two consecutive event_rounds.

### 9. groups.money_visibility_mode v1 vs v1.5

Per epic line 840: `groups.money_visibility_mode TEXT CHECK IN ('open','participant','self_only') DEFAULT 'open'`. v1 only exercises `'open'` (T3-3 UI accepts only open on save; participant/self_only display "v1.5" tooltip + are disabled). The CHECK constraint accepts all three so v1.5 enabling is zero-migration.

## Dev Notes

- **Why one big schema story instead of split-by-domain (events / groups / rules / subgames as separate stories):** the epic intentionally bundles them because (a) every subsequent T3 story consumes 2+ of these tables (T3-2 wizard touches events + event_rounds + groups + invites; T3-5 rule editor touches rule_sets + rule_set_revisions + event_rounds), and (b) splitting would mean N migrations instead of 1, increasing migration risk and making the rollback story unclear. A single additive migration is the simplest delivery.

- **Why `name TEXT NOT NULL DEFAULT ''` rather than NULLABLE:** the architecture wants a non-null name for downstream NOT NULL display logic. Existing T1-6a-bound rows (zero in production, possibly some in dev DBs) get an empty string; T3-2 wizard + T3-3 group CRUD reject empty names at write boundaries.

- **Why `effective_from_hole = 19` for "next round onward":** per epic line 844 + T5.11 spec. Hole 19 is the sentinel value (CHECK is `BETWEEN 1 AND 19` not `BETWEEN 1 AND 18`). Clean approach — no separate boolean column needed.

- **Why FK ON DELETE RESTRICT for course_revisions and players (vs CASCADE):** preserve audit trail. If an event references course_revision X, you should NOT be able to delete X out from under the event (DELETE returns an error instead). RESTRICT is the safe default for "this thing is shared infrastructure with audit value."

- **Why FK ON DELETE CASCADE for event children (event_rounds, groups, group_members, invites, etc.):** these rows are USELESS without their parent event. Cascading deletion is correct semantics + simpler than RESTRICT + manual cleanup.

- **Why `sessionId` is FK ON DELETE SET NULL (not CASCADE):** if a session expires and is deleted, the device_binding should remain (the player still owns the device). The binding's session_id is set to NULL — matching the pre-SSO claim state. T3-7's "post-SSO rebind" flow re-populates session_id when the next session is created.

- **Why integer-cents on `buyInPerParticipant` (not REAL):** mirrors the existing money discipline in Wolf Cup engine (`packages/engine` integer cents) + tournament's planned T6 money matrix. v1 default is 0 ($0); future stories may enable buy-in pots.

- **Wolf Cup isolation (FD-1 / FD-2):** T3-1 writes only under `apps/tournament-api/src/db/schema/**` and `apps/tournament-api/src/db/migrations/**`. Zero edits to `apps/api/**`, `apps/web/**`, `packages/engine/**`.

- **Retro AI-1 applied:** spec codex caps at 4 rounds OR zero-High-zero-Med, whichever first. Same for impl codex.
- **Retro AI-2 applied:** zero SHARED files pre-announced in §1. No gates expected during impl.
