### 3. The two overlapping UNIQUE constraints on `hole_scores`

Per epic AC line 1273-1275, `hole_scores` carries TWO UNIQUE constraints:

- **Cell-level UNIQUE**: `UNIQUE(round_id, player_id, hole_number)` — exactly one score row per cell per round; second INSERT at the same cell with a NEW client_event_id throws.
- **Dedupe-target UNIQUE**: `UNIQUE(round_id, player_id, hole_number, client_event_id)` — used as the `ON CONFLICT (...) DO NOTHING` target so an idempotent replay (identical cell + identical client_event_id) silently succeeds without inserting a duplicate row. The UNIQUE itself doesn't "allow" the duplicate (UNIQUE is UNIQUE) — the production INSERT pattern uses ON CONFLICT against this constraint as the upsert target so the second INSERT is a no-op.

**Codex round-1 raised a real concern**: when both constraints would fire (case (a) below — identical client_event_id replay), SQLite picks ONE of the violated constraints to report. If it picks the cell-level UNIQUE (which is NOT the ON CONFLICT target of the dedupe INSERT), the upsert ABORTS instead of deduping. The fix is to write the INSERT with the conflict-target matching the more specific constraint (the dedupe target), and rely on SQLite preferring to report a constraint that the upsert can handle.

**The exact INSERT form production code MUST use** (T5.6 owns the production path; T5-1's tests must use the SAME form to prove it works):

```ts
await db
  .insert(holeScores)
  .values({ id, roundId, playerId, holeNumber, grossStrokes, putts, scorerPlayerId, clientEventId, ...timestamps, tenantId, contextId })
  .onConflictDoNothing({
    target: [holeScores.roundId, holeScores.playerId, holeScores.holeNumber, holeScores.clientEventId],
  });
```

The drizzle `onConflictDoNothing({ target: [...] })` form emits `ON CONFLICT(round_id, player_id, hole_number, client_event_id) DO NOTHING`. SQLite's documented behavior: when the conflict-target matches one of the violated constraints, the ON CONFLICT clause fires for that constraint. Both constraints fire on identical-replay, but the dedupe target IS one of the violated constraints, so DO NOTHING runs and the row is silently dropped.

When client_event_id differs, ONLY the cell-level UNIQUE fires; the dedupe target is NOT violated; SQLite falls back to default conflict resolution (ABORT) on the cell-level UNIQUE → throws `SQLITE_CONSTRAINT_UNIQUE` (sentinel 2067).

**Epic AC line 1275** asserts: "Codex verified this behavior locally on SQLite 3.50.4 with a dual-constraint repro." T5-1's AC #4 tests are the in-tree repro — they MUST pass on the libsql version we use, otherwise the design is broken for our actual runtime and we fall back to the contingency below.

### 4. `rounds.holes_to_play` mirror semantics

Per epic AC line 1259:

> `rounds.holes_to_play` mirrors `event_rounds.holes_to_play` at round-open time (seeded from the parent event_round); locked once set.

T5-1 ships the column + CHECK constraint (`holes_to_play IN (9, 18)`) + `DEFAULT 18`. **The seed-from-parent + lock semantics are NOT enforced at the schema level**; they're an application-level invariant for T5.8 (round lifecycle state machine) to honor when transitioning out of `not_started`. Schema permits any of (9, 18); doesn't observe the parent.

This is a deliberate v1 simplification: the mirror is documented (in this spec + the column comment) but enforcement lives in T5.8's open-round handler, not the schema.

### 5. `event_id` + `event_round_id` NULLABLE for FD-7 forward compat

Per epic AC line 1259:

> both `event_id` and `event_round_id` are NULLABLE per FD-7 forward compat (standalone-round v1.5 shape); v1 always writes non-null

T5-1 ships both as nullable text FKs. v1 application code (T5.8 round-open handler) MUST always write non-null values. v1.5+ standalone-round flow can leave them NULL when a round exists outside any event (e.g., a casual scorecard captured pre-event-creation).

T5-1 does NOT enforce a "v1 must write non-null" CHECK — that lives in the T5.8 application layer.

### 6. `score_corrections` is append-only — no UPDATE path v1

Per epic AC line 1261:

The schema has no `updated_at` column and no UPDATE-friendly index. T5.9 (the `score_corrections` write route) MUST INSERT only; never UPDATE. T5-1 documents this with a column comment on the table; enforcement is at the T5.9 application layer.

### 7. `round_states` carries CURRENT state only; history goes to `audit_log`

Per epic AC line 1262:

> current state only; historical transitions land in `audit_log` per T5.8

The schema has `round_id` as PRIMARY KEY (not `id`), so each round has at most one row. T5.8's open/transition logic UPSERTS this row and ALSO INSERTS one row per transition into `audit_log` (with the prior + new state). T5-1 ships the table; T5.8 writes the history pairing.

### 8. `audit_log` shape

Per epic source references at lines 1255, 1262, 1465, 1620, 2455, 2586 (audit_log used for state transitions, score commits, install-prompt-shown events), the audit_log table is:

```
audit_log(
  id TEXT PRIMARY KEY,
  event_type TEXT NOT NULL,           -- e.g. 'round.state_changed', 'score.committed', 'install_prompt.shown'
  entity_type TEXT NOT NULL,           -- e.g. 'round', 'hole_score', 'session'
  entity_id TEXT NOT NULL,             -- the record id (text per FD-6)
  actor_player_id TEXT NULL FK → players.id,  -- NULL for system events
  payload_json TEXT NOT NULL,          -- bag of event-specific data
  created_at INTEGER NOT NULL,
  + ecosystemColumns()
)
```

Indexes:
- `idx_audit_log_entity_type_id` on `(entity_type, entity_id, created_at DESC)` — primary read pattern: "show audit history for this entity"
- `idx_audit_log_event_type_created_at` on `(event_type, created_at DESC)` — secondary: "show all state transitions in last 24h"

The `audit_log` table is **NOT FK'd to specific entities** — `entity_type` + `entity_id` is a string-id polymorphic association, intentional so we can audit deletes (the entity row may be gone). FK on `actor_player_id` IS enforced (RESTRICT) since players are shared infrastructure.

T5-1 ships the table; T5.8 / T5.9 / T7-6 wire the writes.

### 10. FK delete posture

## Risks

- **Two UNIQUE indexes on `hole_scores` interaction** — non-trivial. SQLite evaluates both on every INSERT; we rely on (a) the dedupe target succeeding via `ON CONFLICT(round_id, player_id, hole_number, client_event_id) DO NOTHING` when client_event_id matches, and (b) the cell-level UNIQUE throwing 2067 when client_event_id differs. AC #4's three tests are load-bearing — they MUST pin this behavior at every commit, especially if drizzle-kit ever changes how it emits UNIQUE indexes (e.g., switches to a single composite index strategy). Codex round-1 spec review flagged this as a High because SQLite's choice of which constraint to report when multiple fire is implementation-defined; epic AC line 1275 asserts codex verified the design on SQLite 3.50.4. AC #4's Test 4a is the in-tree repro on our libsql version. **Contingency** if 4a fails: drop the dedupe UNIQUE, push idempotency to T5.6 app code (SELECT-by-cell-then-INSERT-or-return-200 in a transaction). See Risk Acceptance §3.

- **`event_id` + `event_round_id` NULLABLE** — by design (FD-7). v1 application code must always write non-null. Risk: a v1 bug writing NULL would silently succeed at the schema layer; T5.8's open-round handler must guard. NOT a T5-1 concern, but documented.

- **`event_id` + `event_round_id` inter-FK consistency** — by design we do NOT enforce that (event_id IS NULL) ⇔ (event_round_id IS NULL), nor that `event_id` matches `event_rounds.event_id` when both are set. Codex round-1 flagged this as a Med. Rationale for deferring: the consistency invariant lives at T5.8's open-round handler (it ALWAYS seeds both from a single source — the parent event_round — and never permits independent NULL-state combinations in v1). Schema-level enforcement would require either a CHECK constraint (works for the simple "both NULL or both set" case) or a trigger (for the parent-match invariant). The dev SHOULD add a CHECK: `((event_id IS NULL) = (event_round_id IS NULL))` to the `rounds` table since it's a one-line additive guard with zero cost; this catches accidental NULL/non-NULL split states without touching trigger semantics. Skip the parent-match trigger (over-engineering for v1).

- **`audit_log` polymorphic association (`entity_type` + `entity_id`)** — by design. NOT FK'd to specific entities, so a deleted entity's audit rows survive (intentional). Risk: a typo in `entity_type` (e.g. 'round' vs 'rounds') silently fragments the audit history. T5.8 / T5.9 / T7-6 callers must use a shared constant. NOT a T5-1 concern, but documented.

- **`score_corrections` append-only NOT enforced via SQLite triggers** — by design. Codex round-1 flagged this as a Med (triggers ARE possible: `CREATE TRIGGER ... BEFORE UPDATE/DELETE ON score_corrections RAISE(ABORT, 'append-only')`). T5-1 deliberately defers trigger enforcement: (a) the schema is already minimal — adding triggers means more migration SQL surface for ZERO production benefit (no app code is going to UPDATE corrections), (b) drizzle-kit's trigger generation is opt-in / non-standard and would fragment the migration story, (c) the T5.9 spec OWNS the INSERT-only invariant and will pin it via test (one INSERT path; no UPDATE path exists). If a future dev forgets and adds an UPDATE handler, code review + the T5.9 test surface should catch it. Document this choice + the trigger-based fallback in the `score_corrections` column-comment block in `scoring.ts`.

- **Tenant scoping is column-level, NOT in FKs / UNIQUE constraints** — Codex round-1 flagged this as a Med. By design: tournament v1 is single-tenant ('guyan' default on `tenant_id`); cross-tenant reference is a theoretical concern, not a v1 risk. Application-layer enforcement is the actual security boundary: every route filters every SELECT/UPDATE/DELETE on `tenant_id = TENANT_ID` (post-T3-7 hardening pattern, established across T3-7..T4-3). T5-1 inherits this pattern; downstream T5 routes will too. Restructuring FKs to be tenant-aware (composite `(tenant_id, id)` PKs + composite FKs) would require a much larger refactor across T2-T4 schemas — out of scope for v1. Document the limitation explicitly so future multi-tenant work knows where to start.

- **`score_corrections.request_id` has NO uniqueness constraint** — Codex round-1 flagged this as a Low. T5.9 is the owner. T5-1's role: ship the column NULLABLE-FALSE TEXT. Whether T5.9 wants it as an idempotency key (UNIQUE) or just a diagnostic correlation token (no constraint) is a T5.9 design decision. Document this deferral in the `score_corrections` column-comment block in `scoring.ts`.
