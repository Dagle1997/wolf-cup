## 📊 Mary (Analyst) — Strategic / Threat-Model Perspective

T3-6 unlocks the FR-E1 (revised 2026-04-18) "zero-friction first arrival" promise. Strategic significance: a Pinehurst Crew player who taps `tournament.dagle.cloud/invite/<token>` from a GroupMe message gets straight to a roster picker — no sign-in page, no SSO redirect, no account creation. They tap their name, the device is registered, and they're "in." This is the inflection point where tournament-api's 6-week schema-and-admin work becomes player-visible.

**Threat model — five surfaces worth scrutinizing:**

1. **Token brute-force enumeration.** The invite token is generated by T3-2 via `crypto.randomBytes(32).toString('base64url')` → 256 bits → 43 chars. Brute force at 10M guesses/sec would take ~10^60 years. **Bulletproof.** No rate limiting needed at v1.

2. **Cookie sharing → impersonation.** The device cookie value IS the credential for the bound device_binding. Sharing the cookie = full impersonation of that player_id (until SSO via T3-7). Acknowledged v1 limitation. Mitigations:
   - HttpOnly attribute prevents JS access (XSS can't lift the cookie value).
   - SameSite=Lax prevents cross-origin POSTs from triggering claims.
   - The blast radius is bounded: no mutation surface for unauthenticated devices. T5 score entry, T7 photo upload, admin actions all REQUIRE SSO. The compromised device can VIEW event data + re-tap a name, nothing more.
   - Future hardening: signed cookies or per-device tokens with rotation. Out of v1 scope.

3. **Cross-event UPDATE protection.** Without the `existingRow.contextId === 'event:{currentEventId}'` guard (added at R2 codex), a returning device-cookie holder following an invite for a NEW event would corrupt the original device_binding's event-scope semantics. The guard is load-bearing for correct multi-event behavior. Tested via the explicit cross-event test pinning the orphan-row outcome.

4. **Multi-device same-player.** One player's spouse + phone + tablet all click the same invite. v1 allows independent device_bindings per device. T3-7's post-SSO rebind will consolidate when the player completes SSO on one device. **Acceptable; documented.**

5. **Cookie expiration drift on the 90-day Max-Age.** Pinehurst is 4 days. 90 days is generous overkill but harmless. A returning player a year later still has a stale binding pointing at last year's event_id; their next invite click for a new event INSERTs a fresh row (cross-event protection fires). **No real-world risk; the cookie length serves the "I checked the schedule on Tuesday and want to check again Wednesday" UX, not the long-term arc.**

**One v1 limitation worth flagging:** the success-surface "Event schedule (coming soon)" link goes nowhere because T7 hasn't shipped. AC #16 manual smoke notes this explicitly. **Document at deploy time** so the Pinehurst Crew doesn't get confused on a dead link.

**Recommendation: ship.** AC #16 manual smoke is the final gate.

## 🏗️ Winston (Architect) — System Design Perspective

Six observations:

1. **5th `/api/admin` mount avoided.** T3-6 routes live at `/api/invites` — a NEW prefix. Winston's "promote umbrella adminRouter at ~5 mounts" threshold from T3-3/T3-5 reviews doesn't tip. The `/api/admin` umbrella refactor stays a future story (T3-7 may push it; T3-8 permissions middleware almost certainly will). **Hold.**

2. **2nd cookie-builder pattern.** `session.ts:177-197` (`sessionCookieHeader`) and `invites.ts` (`deviceCookieHeader`) are now siblings. Each has slightly different defaults: session uses SameSite=Strict + Domain attribute + 7-day Max-Age; device uses SameSite=Lax + no Domain + 90-day Max-Age. The shared logic is the conditional Secure-on-production check. A generic `buildCookieHeader(name, value, opts)` helper could promote, but the differences are load-bearing per-cookie + the helpers are 20 lines each. **Hold; promote when 3rd consumer arrives.**

3. **5th copy of `extractCookie`** (well, ~3rd — auth.ts has its own; require-session.ts has its own; invites.ts now has its own; admin-courses.ts and admin-rule-sets.ts don't extract cookies but read them via middleware). The function is 10 lines. **Same hold pattern; the codebase posture explicitly tolerates these duplications. Promote at 4th-extractor consumer to a shared `lib/cookie-utils.ts`.**

4. **The cross-event UPDATE guard** (`existingRows[0].contextId === expectedContextId`) is the correctness centerpiece of T3-6. Without it, the same device cookie following invites for two different events would corrupt the event-scoping semantics. The guard is simple — one string comparison — but the consequence of getting it wrong (silent cross-event data leakage) was significant. **Spec round-2 codex caught it; impl pins it via test. Solid.**

5. **The PUBLIC route pattern** (no `beforeLoad`) is novel in tournament-web. Future stories that need similar anonymous-friendly routes (T7's read-only event view? T3-10's GHIN profile? — actually no, T3-10 needs auth) will follow this template. Document the absence of `beforeLoad` as a deliberate choice (already done via comment "NO beforeLoad — anonymous-friendly per FR-E1"). **Pattern set for future PUBLIC routes.**

6. **TanStack Query `useQuery` with `retry: false`** in the inviteQuery + the 5xx error-state surface is the right shape. A 4xx (404 / 410) shouldn't auto-retry; a 5xx might be transient but auto-retry would mask the issue. The user-facing "try again in a moment" message is enough. **Idiomatic.**

**Architectural concerns: zero blockers.** Three "watch and promote" notes (umbrella adminRouter, cookie-builder helper, cookie-utils lib) — none warrant T3-6 changes.

**Recommendation: ship.**

## 📋 John (PM) — User Value / Scope Perspective

**Does T3-6 satisfy organizer-facing value?** Yes — but the value is REVERSED here vs prior T3 stories. Most T3 stories shipped organizer admin tools (T3-2 wizard, T3-3 group CRUD, T3-5 rule editor); T3-6 is the FIRST player-facing surface. Pinehurst Crew members tap a link from GroupMe; no setup-day wrangling about "did everyone create an account."

**AC #16 manual smoke walkthrough is the load-bearing test:**

1. **Open invite link from messaging app on phone.** SameSite=Lax must allow the cookie to be sent on the top-level navigation. Lax was deliberate (vs Strict which would suppress the cookie on the very first invite click). Smoke must verify on at least one mobile browser.

2. **Roster picker renders with all 8 Pinehurst players.** If T3-3's group_members were populated correctly, the names show. If T3-3 left a player out (typo, etc.), they're not in the roster. **Smoke step: cross-check roster count vs Josh's list of crew members.**

3. **Tap a name → success surface.** No 5xx, no 4xx. The Set-Cookie header lands; future invite-page reload still shows "you're in" (would require the device cookie to flow). Smoke step: reload the page anonymously after the claim — does the user see the success surface? **Actually the current spec doesn't cover this** (the GET /:token endpoint doesn't read the device cookie or behave differently for already-claimed devices). The reload behavior shows the picker again. v1 minor friction but not a blocker.

4. **Expired token error.** Smoke step: DB-edit `expires_at` to past + reload page → "this invite has expired" message.

5. **Device cookie inspection in DevTools.** Smoke step: confirm `tournament_device_id` cookie set with `HttpOnly`, `SameSite=Lax`, `Secure` (in production), `Path=/`, `Max-Age=7776000`, no Domain attribute.

**Scope discipline: tight.**
- Zero new deps. TanStack Query already in use.
- ZERO SHARED gates. Eighth tournament story without SHARED (T3-4 was the only one).
- Spec went 4 rounds in codex (hitting the AI-1 cap) — heavier than T3-5 (3 rounds) but lighter than T3-3 (4 rounds with multi-issue R1). The cookie semantics + cross-event protection drove most of the iteration.
- Impl-codex returned ZERO findings on R1. **Best impl-codex outcome to date.** The thorough spec paid off.
- 15 backend tests + 5 frontend tests. Both exceed minimums.

**Concerns / observations:**

1. **Schedule link goes nowhere v1.** Confirmed acceptable in spec. Future T7 lands the destination.

2. **No re-claim UX for already-claimed devices.** A returning player on the same device sees the picker AGAIN, not a "welcome back" surface. They tap their name (or someone else's), UPDATE branch fires, they see success. Mild friction. **Future polish: GET /:token reads the device cookie + returns the bound player_id; UI shows "welcome back, Alice" with a small "actually I'm someone else" link.**

3. **The "ask Josh for a new invite" copy** is hardcoded. Future polish: parameterize with the organizer name from the event row.

4. **No empty-state UX if roster is empty.** Tested — the UI shows "The event roster is empty. Ask Josh to add players first." Acceptable v1 corner case.

5. **No throttling on POST claim.** A bad actor could brute-force player names from a stolen token. v1 closed-roster bounds this to 8 attempts per token. Future hardening: per-token rate-limit (e.g., 10 claims/minute).

**Recommendation: ship.** AC #16 manual smoke at `tournament.dagle.cloud/invite/<token>` from a phone is the final gate.

## 💻 Amelia (Dev) — Code Quality / Maintainability Perspective

Six observations:

1. **`deviceCookieHeader` builder is 14 lines** — short, obviously correct, deliberately mirrors session.ts's pattern (lines 177-197). Conditional Secure on `env.NODE_ENV === 'production'` matches the established posture. Comment cites the source. **Idiomatic.**

2. **The cross-event UPDATE guard** at admin-rule-sets... wait, T3-6's invites.ts. The guard is one line: `existingRows[0]!.contextId === expectedContextId`. Fall-through to INSERT branch happens via a `let updated = false` flag that flips to true only when the UPDATE actually fires. Clean state machine; no early-return mess. **Readable.**

3. **The `extractCookie` helper** is duplicated. Auth.ts has it. require-session.ts has it. invites.ts now has it. Per the codebase comment in auth.ts:528-534, the next consumer should promote to a shared util. T3-6 IS that next consumer numerically (3rd usage), but the deduplication threshold has historically been "4th consumer" per Winston's notes. **Hold; the codebase isn't tightening the rule.**

4. **The `let updated = false` + post-block check pattern** in the POST claim handler keeps the cookie-aware branching readable. An alternative early-return-everywhere structure would be shorter but harder to follow when you're tracing the cookie + cross-event guard through. **Acceptable verbose-but-clear.**

5. **No `// eslint-disable`, no `as any`, no implicit any.** Casts limited to:
   - `(input as Request).url` in test mock router — narrowing fetch's RequestInfo union.
   - `(await res.json().catch(() => null)) as ClaimResponse & ClaimError | null` — narrowing fetch's untyped JSON.
   - Error narrowing in mutationFn catch (`new Error() as Error & { code?: string }`).

6. **The `if (code !== undefined) err.code = code;` pattern** (vs `err.code = body?.code ?? undefined`) is the explicit defense-in-depth against `exactOptionalPropertyTypes: true`. The simpler `err.code = body?.code` would assign `undefined` to a property typed as optional-but-not-undefined. Strict TS configuration catches this; the impl was forced into the explicit form by the typecheck pass. **Correct; no shortcut.**

**Mid-impl observations:**

- **First impl pass had a `let roster` that ESLint flagged as "should be const"** — caught by lint pass (1-line fix). Standard codebase strictness.
- **First impl pass had a `body?.code` direct assignment that strict TS rejected** — caught by typecheck. The fix is one of the canonical patterns for exactOptionalPropertyTypes.
- **Spec round-2 codex caught the cross-event UPDATE corruption** before impl ever wrote it — the spec-first iteration genuinely paid off here. The impl came in at zero codex findings on round 1.

**Two minor cleanup items (Lows):**

- The `releaseController(ac)` in the mutationFn `finally` block runs even when the fetch was aborted via unmount. Slight inefficiency (the Set has the controller, releaseController removes it; aborted controllers don't need explicit removal because the useEffect cleanup will handle them on unmount). Trivial; future polish.
- The "Welcome, X! / Event: Y" success surface uses a hardcoded "Event:" label. Future polish: design pass for the success state.

**No blockers.** Ship.
